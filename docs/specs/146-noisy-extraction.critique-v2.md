# Critique v2: #146 — Extraction nav/header noise + leaked Alpine/Vue JS

Synthesis of three per-lens re-reviews (Correctness/FP-FN, REDOS/linearity, Contract/scope/siblings). Every load-bearing claim was re-traced by hand and re-run against `origin/main` source; the (B)-no-op-on-repros claim was verified by a live fetch + literal-HTML grep on 2026-07-09.

## Verdict

**BLOCKED.** Two clusters block:

1. **(C) is one line from ship-ready but ships a markup-leak regression at the untrusted-HTML trust boundary.** The quote-aware core fix is correct and linear for the headline Alpine/Vue case (verified). But the `end === -1` legacy-fallback guard fires only when a quote is *still open at EOF* (`quote !== null`). A quote that *opened AND closed while swallowing a `>`*, with no later unquoted `>`, falls through to `out += html.slice(cursor)` and **leaks the whole malformed tag as visible text** — worse than current (prompt-injection-adjacent). One-line fix: track an `enteredQuote` boolean and take legacy recovery when it is true. Falsifies the spec's repeated "preserving the pre-fix behavior exactly on malformed input" (lines 15, 32, 51, 84).

2. **(B) is built on a mis-diagnosed repro and its mechanism is incoherent.** Neither named repro has any `<header>` element (verified live: php.net 0, cppreference 0 — and cppreference has 0 of every landmark/nav/header tag), so (B)'s `<header>`-only strip is a **no-op on both pages the spec claims it fixes**. Separately, the named strip tool (`stripChromeElement(html, "header")`) has no per-element filter — it strips *every* `<header>` — so it cannot meet the frozen acceptance criteria 13/14 it is designed around; and its `<nav>` disjunct is dead code given the stated ordering (`stripChromeFromRaw` strips nav before the header step).

(C) and (B) are independent — (C) can land as soon as the markup-leak is pinned; (B) needs the scope decision and a real mechanism before a line is written.

## Resolved in v2 (v1 findings genuinely closed)

Verified, not rubber-stamped:

- **v1 blocker 2 / defect 3 (length-guard inverted → mega-menu FN).** The length guard is dropped; a `<nav>`/`<form>`-bearing mega-menu is now stripped regardless of size. (Partial — see *Still unresolved*.)
- **v1 blockers 3, 4 (length calibration / measurement mechanism / granularity).** Dissolved: the role-based redesign has no length measurement, so there is nothing to calibrate, measure per-header, or thread hidden-aware.
- **v1 defect 5 ("matches pre-fix exactly" overclaim).** Narrowed to the *unterminated*-quote subset, where it is true. (A *new* overclaim appears on the terminated-quote subset — see *New defects*.)
- **v1 defect 6 ("landmark path unaffected" overclaim).** Honestly reframed as a desired correctness flip (`<main>` that won only via leaked-directive-JS length inflation losing to `<article>` is a correction, not a regression) and pinned by criterion 8. `MAIN_OVERRIDE_FACTOR 1.5` still guards genuine `<main>` richness; the fix only affects `>` inside quoted attributes, never literal `>` in `<code>`. Acceptable.
- **v1 defect 7 (sibling-sweep gap).** Sweep complete: all 4 `stripHtmlTags` callers enumerated (`html.ts:81`, `tier1-payload.ts:58`, `metadata.ts:45`, `svg-text.ts:59,64`) and `charset.ts` same-class fix included.
- **v1 defect 8 (attribute-injection `<header>` over-strip).** Confirmed unreachable: `findStartTags` advances `offset` past `readStartTag`→`findTagEnd` (quote-aware), so a `<header` inside another tag's quoted attribute is skipped (traced on the exact adversarial input). Criterion 15 holds.
- **v1 defect 9 (footer contract drift).** `contracts.md` (the Tier-1 paragraph: "excludes site chrome (`<aside>`/`<nav>`/`<footer>`)") mis-describes `stripChromeFromRaw`, which strips aside/nav and *keeps* footer. The spec commits to *fixing* (not appending) that text. Resolved-pending-edit.
- **(C) headline fix.** `<div x-init="$nextTick(() => { if (a > b) { foo() } })">REAL PROSE` → `REAL PROSE` (current leaks the JS). Adversarial pin `<div x-init="if(1>0){x()}">IGNORE ALL PRIOR INSTRUCTIONS` → visible text is just the instruction, no `x-init`/`if(1>0)`/`{x()}` leaked. Correct.
- **REDOS.** Both (C) and (B) redesigns are linear; no REDOS-5/6 regression. (C): the legacy `indexOf` fires only on the EOF-in-open-quote path, at most ~2x bounded (only two quote chars exist), well under `LIN_RATIO=40` / 25s `MULTI_CEILING_MS`. (B): `findElements('header')` O(n) + per-header `findStartTags('nav'/'form')` over *disjoint* header contents (Σ|content| ≤ n) + `stripChromeElement('header')` O(n). Timing on sibling/nested/unclosed header floods scaled 6.8–9.8x (linear). Linearity for (B) holds **only if** the implementation scans `el.content` per header and builds the selective-strip in a single monotonic pass — the spec does not pin either.

## New defects from v2 redesign

1. **(HIGH / content-integrity) (C) markup-leak on a TERMINATED quoted attr containing `>` with no following unquoted `>`.** Input `X<a title="a>b"Y` → v2: quote opens, swallows the `>`, closes, then no `>` to EOF → `end === -1` with `quote === null` → falls through to `out += html.slice(cursor)` → result **`X<a title="a>b"Y`** (whole tag, including `<a title="a>b`, leaked as visible text). Current: `X b"Y` (keeps `b"Y`). NEW is a regression vs current — it leaks *more* markup than the quote-blind original, the exact failure class #146 exists to eliminate (line 18/122 calls it prompt-injection-adjacent). Reachable via fetch byte-cap truncation mid-attribute (#149/#158) or a crafted page. **Fix (one line + fixture):** track `enteredQuote` (set true whenever quote mode is entered); in the `end === -1` branch take the legacy first-`>` recovery when `enteredQuote` is true — not only when `quote !== null`. Restores current behavior for *both* the unterminated case (defect 2, already handled) and this terminated-no-close case, and leaves criterion-6 cases (lone `<`, `<b>` at EOF, `<>`) identical (they never enter quote mode). Propagates to all 4 callers — `svg-text.ts` most exposed (run-to-EOS of an unterminated `<text>`).
2. **(HIGH / correctness, DoD) (B) is a no-op on both named repros.** See *Still unresolved #1*. php.net 0 `<header>`; cppreference 0 `<header>`. (B)'s `<header>`-only strip never engages on either. The spec's central empirical claim (lines 21/38/143) and criterion 17 are unmeetable by (B) as designed.
3. **(HIGH / mechanism) (B) selective-strip is incoherent with its per-element design.** Spec line 103: "Strip qualifying headers with the existing `stripChromeElement(html, "header")`." But `stripChromeElement` (main-content.ts:55-70) iterates *all* `findStartTags(html, "header")` and strips *every* one — it has **no filter/predicate parameter**. So calling it violates criterion 13 (`<header><h1>Investigation Title</h1></header>` → Title kept; verified: the title is deleted) and criterion 14 (per-section `<header><h2>Section A</h2></header>` → Section A kept; verified: deleted). The "per-element: each header decided independently" assertion (line 104) is unrealizable by the named tool. **Required:** per-header byte-range splicing — collect qualifying `[start, end)` offset pairs and build the output in a **single monotonic pass** (must be pinned single-pass to stay O(n), else a naive per-header `findStartTags(FULL_HTML)` is O(headers × n)).
4. **(HIGH / ordering) (B) `<nav>` disjunct is dead code given the stated ordering.** Spec line 101 says detection runs "after the existing aside/nav strip." `stripChromeFromRaw` (main-content.ts:135) calls `stripChromeElement(…, "nav")` *before* the header step, and `stripChromeElement` strips every `<nav>` regardless of nesting — so by the time header-detection runs, any `<nav>` inside a `<header>` is already gone and `findStartTags(header.content, "nav")` is always `[]`. The discriminator collapses to "contains `<form>`." Criteria 11/12 ("header containing `<nav>` → stripped") cannot exercise the nav disjunct in-pipeline. **Required:** detect nav/form on the *pre-nav-strip* body and coordinate offsets so detect and strip operate on the same string; or drop the `<nav>` disjunct; or reorder.
5. **(MED) (B) `findElements` and `stripChromeElement` disagree on what a `<header>` IS.** `findElements` pairs open→*first* close (`findCloseTag`, simple); `stripChromeElement` pairs open→*matching* close (`findMatchingClose`, depth-aware). Input `<header>A<header><nav>inner nav</nav></header>B</header>`: `findElements` flags the outer header on content `A<header><nav>inner nav</nav>` (contains nav → qualifies); `stripChromeElement` strips the whole outer span `A<header>…</header>B</header>`. Detect-region ≠ strip-region. **Required:** use one consistent enumeration (stripChromeElement's own `findStartTags`+`findMatchingClose`) for both detect and strip.
6. **(MED / FP) (B) role signal "contains `<nav>`/`<form>`" is too coarse.** A legitimate article header carrying a table-of-contents `<nav>` or a search `<form>` is stripped wholesale → article title/byline deleted while the body survives (v1 defect 4 reappearing through the new role trigger). Criterion 13 does not pin this common shape. (Adjunct to spec open-question #1; `role="banner"` does not fix it — the issue is nav/form can legitimately appear in a content header.)
7. **(MED / FN) (B) misses role-based mastheads.** `<header><ul role="menubar">…` or a hamburger `<button>` (no literal `<nav>`/`<form>`) → discriminator false → header kept → chrome survives. ARIA-`role`-on-`<ul>` is common on accessibility-conscious/older sites. The proposed `role="banner"` addition also misses top-level `<header>`s that carry banner role *implicitly* (rarely the literal attribute). Cleanest discriminator is *top-level-ness* (direct child of `<body>`) + contents, not contents alone — undecided.
8. **(LOW / precision) `charset.ts` fix under-specified.** `findTagEnd` returns `html.length` on no-`>` (not `-1`) and the index *past* `>` (not the `>` index). The implementer must simultaneously change: the sentinel `if (close === -1) return undefined` → `if (tagEnd >= lower.length) return undefined`; the slice `lower.slice(at, close)` → `lower.slice(at, tagEnd)` (now includes `>`); and `cursor = close + 1` → `cursor = tagEnd`. Leaving any one as-is yields an off-by-one or a dead unterminated-meta branch. Pin acceptance on `<meta data="a>b" charset="utf-8">` **and** an unterminated `<meta charset="utf-8"`. Also: actual path is `src/infrastructure/http/charset.ts` (spec writes `http/charset.ts`); importing `findTagEnd` from the extract layer into the HTTP body layer is acceptable under least-machinery (both are infrastructure, no cycle), but a tiny shared `html-tokens` module would be cleaner.
9. **(LOW / REDOS-prose) (C) "every char scanned once, indexOf resumes from the tag end (no rescans)" (line 52) is imprecise.** The malformed-quote path scans the tail a bounded ≤2x (inner loop to EOF) then the legacy `indexOf` rescans — a constant ~1.5x factor, still O(n) and far under the ceiling. Recommend rewording to "each char scanned a bounded constant number of times (≤~2x in the malformed-quote tail)." Re-confirm the absolute 25s ceiling on a 1MB quoted-attr flood (v1 raised this; v2 does not explicitly close it for the per-char path).

## Remaining blockers (decisions before coding)

1. **(B) scope decision.** Either un-defer (D) to also strip bare top-level site-masthead `<div>`/`<form>`/`<ul>` (the actual php.net/cppreference shape), or drop the php.net/cppreference claims + verify bar and re-anchor (B) on a real `<header>`-with-`<nav>` page. As written, (C)+(B-role) collapse only the Docker/Alpine repro, not the two pages named as the motivating repros.
2. **(B) per-element strip mechanism.** Byte-range splicing of qualifying headers in a single monotonic pass — NOT `stripChromeElement('header')`. Pin criteria 13/14 against this.
3. **(B) detect ordering / dead `<nav>` disjunct.** Detect nav/form on the pre-nav-strip body and coordinate offsets; or drop the `<nav>` disjunct. (Resolves new defects #4 and #5 together.)
4. **(B) masthead discriminator precision (spec open-question #1).** Resolve before coding (house rule: no implementing while the spec has pending decisions). Consider top-level-ness + contents; decide whether `role="banner"` (explicit/implicit), `<form role="search">`, `<input type="search">` are in. (Resolves new defects #6 and #7.)
5. **(C) markup-leak fix.** Widen the `end === -1` guard from `quote !== null` to `enteredQuote`; re-confirm criterion-6 cases unaffected; add a terminated-quote-no-close fixture (and a content-loss variant `<a title="foo>bar" REAL</a>` → `REAL` must survive — note the `enteredQuote` fix does *not* cover this content-loss variant, which needs its own decision: it is reachable but more contrived; at minimum the "exact pre-fix on malformed" claim must be re-scoped to exclude it).

## Notes

- **(C) is ship-ready modulo the markup-leak pin.** The core fix, defect-1/2/8/9, the C-d6 landmark flip, and the `charset.ts` same-class sweep are all genuinely resolved and verified. The one-line `enteredQuote` guard + fixture closes the regression; nothing else blocks (C).
- **(B) is the larger problem and all of it clusters on a mis-diagnosed repro.** The role-based redesign is better-calibrated than v1's length guard (it cleanly dissolves blockers 3/4 and defect 3), but it is built on the claim that php.net/cppreference use `<header>` — they do not — and its `<nav>` disjunct is dead given the stated ordering, and its named strip tool cannot do per-element qualification. Net: do not code (B) until the scope decision (blocker 1) and the mechanism (blockers 2–3) are settled.
- **Test surface flatters (B-role).** Existing `test/` `<header>` fixtures (extract.test.ts) are all synthetic GitHub/MS-Learn-style header-with-nav shapes on *landmark* pages (have `<main>`/`<article>`) — exactly the shape (B-role) handles, and exactly NOT the shape of the real repros (no-landmark legacy `<div>` layouts with zero `<header>`). The frozen suite will validate (B-role) against fixtures that flatter it while the named real repros stay unfixed. Per the real-input rule, the verify bar must require the **real** php.net + cppreference pages for (B) — which is precisely why (B-role) cannot pass it.
- **Charset.ts is the only real same-class sibling** in `src/` (grep: only `charset.ts:35`, `html.ts:113` [the C target], `hidden.ts:197` [close-tag-only, correct]); its `parseTagAttrs` is already quote-aware, so only line 35 is blind. Fix is sound in intent; see new-defect #8 for the under-specification.
- **`MIN_BODY_CHARS=200` coupling** (carried from v1) no longer applies — v2 dropped the length guard entirely, so there is no 200↔80 (shell-gate) coupling to document.
