# Critique: #153 — Extract: clearer schema-validation error + fail-fast at input

Independent synthesis of three per-lens reviews (correctness-security, contract-scope, siblings-tests). All empirical claims below were re-traced against `origin/main` source: the walker's recursion set was diffed against `validateAt`'s actual traversal (`json-schema.ts` + `json-schema-composites.ts`); the `provider:"none"`-only-on-degrade invariant was checked at all four writers; the status-classification latent bug was reproduced against the current reason allowlist.

## Verdict

**BLOCKED** (1 blocker, 1 P1). The design's security-critical claims hold up: the new walker is a verified **sound superset** of every subschema location `validateAt` visits (`properties`/`items`/`additionalProperties`-when-schema/`allOf`/`anyOf`/`oneOf`/`not`, plus `$defs`/`definitions`), so no unsupported keyword can slip through to `finalize`; the message never echoes a schema value; fail-closed is correct (both `normalizeCaptatumInput` and `normalizeBulkInput` throw `CaptatumInputError` *before* egress, wrapped by `callCaptatum`/`callBulk` → `toMcpError` → `InvalidParams`); `contracts.md` already reflects the #153 deltas and is internally consistent; and the status simplification is a genuine **conformance fix** for a real latent bug (`contracts.md:484` already mandates `provider:"none"` ⇒ `partial`, so `no_model_fit`/`provider_unconfigured`/`model_unavailable`/`sensitive_content_no_local_provider` degrades are today mislabeled `status:"pass"`). The contract is ready.

The block is the one trust-boundary defect all three lenses independently surfaced (splitting only on severity): the new input-boundary walker ships with **no depth cap**, justified by a **factually false** claim that the request-body *size* limit bounds nesting *depth*. It does not — body size bounds total nodes, a ~1 MB body of `{"a":{"a":{…}}}` reaches ~150K depth and overflows V8's ~10K-frame stack long before the body limit bites. The walker runs **pre-fetch, free to attack** (no egress/LLM cost), so it is a *strictly cheaper* DoS vector than anything shipped today (`validateAt` is post-fetch, egress-rate-limited) — on a tier-S security product. Trivially fixable (depth cap 64–128, or iterative); does not unblock until the no-cap decision is re-made with correct facts.

The P1: the spec claims the `string`→`TransformReason` narrowing is "verified-clean" and lists a complete sibling sweep, but the design invalidates **at least three** existing tests it never enumerates (two `ModelPick` fakes with non-union reason values, plus a `failed`→`transform_failed` rename assertion). A coder following the spec literally hits unexpected red tests.

## Findings & risks (ranked severity)

### Blocker

1. **(BLOCKER / DoS at trust boundary) `findUnsupportedSchemaKeyword` recurses with no depth cap, justified by a false depth bound.** The new walker recurses on caller-supplied (untrusted) schema at the *input* boundary. Spec lines 122–129 and `threat-model.md:120–122` justify the no-cap decision with "bounded upstream by the request-body size limit + `JSON.parse`." Body size bounds **total nodes**, not **depth**: a 1 MB body of nested single-property objects (`{"a":{"a":{…}}}`, ~6 bytes/level) reaches ~150K depth within the 5 MiB body limit (`config.ts:10`), far exceeding Node's ~10K-frame stack ⇒ `RangeError: Maximum call stack size exceeded` ⇒ process crash. The walker runs in `normalizeCaptatumInput` (`captatum-input.ts:104`, called at `captatum.ts:65`) **before** tier-2 (`:68`) and fetch (`:87`) on every `output:"extract"` call, with **no egress/LLM cost** to the attacker — so it is a *strictly cheaper* DoS vector than the cited `validateAt` precedent (`json-schema.ts:28`, reachable only post-fetch via `finalize.ts:49`). "Matching the `validateAt` precedent" under-protects: the precedent is the *less* exposed path and deserves the weaker guard, not parity. The schema field has no depth/size cap at the zod layer (`captatum-input.ts:19` `schema: z.unknown().optional()`).
   - *Severity split across lenses:* contract-scope rated this a **blocker** (free-reachable crash on a tier-S product); siblings-tests rated **P2** (JSON.parse bounds it today, but "most likely to draw review fire"); correctness-security rated **nit** (RangeError is caught by `callCaptatum`/`callBulk` try/catch → `InternalError`, so no crash *vector*, only a mis-stated justification). The aggregate adopts the blocker reading: the residual is real and the no-cap decision was never soundly made.
   - **Resolution:** Add an explicit depth cap (64–128, far above any legitimate schema) to the walker, or implement it iteratively. Re-make the no-cap decision with correct facts (body size does not bound depth). Cheap defense-in-depth at a high-frequency trust boundary; not a least-machinery violation. Correct the spec + `threat-model.md` wording to name V8's JSON.parse recursion limit (not the body-size limit) as today's implicit depth bound.

### P1

2. **(P1 / test-ripple + sibling-sweep completeness) The spec claims the `TransformReason` narrowing is verified-clean and the sweep complete; it is neither.** The spec's Type-ripple § (lines 173–175) and Sibling-sweep § (line 235) claim "only ever set from `noneReason`," "no new values invented," and "Grep confirms no other `transform.reason` constructor or `reason ===` reader outside the above." That holds for **production** but not for **tests** — the verification was never run. At least three existing tests break, unenumerated:
   - `test/finalize.test.ts:10` — `RecordingRouter` fake returns `{ provider: "none", reason: "test" }` from a `ModelPick`-typed fake; `"test"` is not a `TransformReason` ⇒ type error after the `ModelPick.reason` narrowing (`model-router.ts:38`).
   - `test/mcp-shape.test.ts:96,105` — `base()` passes `transform:{ provider:"openrouter", …, reason:"selected" }` into a strictly-typed `Result`; `"selected"` is not a `TransformReason`.
   - `test/llm-transform.test.ts:264` — asserts `deepEqual(result.transform, {…, reason:"failed", latencyMs:4})`; the rename maps this catch path (`captatum.ts:174`) to `reason:"transform_failed"`.
   - Additionally, the fail-fast-at-input change (design 2) **invalidates two whole test bodies**, not just an assertion: `test/llm-transform.test.ts:163–186` (`output:"extract"`, schema with `format`) and `:188–215` (nested `anyOf`/`oneOf`/`not`) currently assert a *returned* `{output:"raw", errors:[{code:"extract_schema_invalid", …}]}`. After design (2), `normalizeCaptatumInput` throws `CaptatumInputError` at `captatum.ts:65` *before* the fetch at `:87`, so `execute()` throws instead of returning — both message and code change (`extract_schema_unsupported_keyword`). Coverage of the *retained* `finalize` defense-in-depth branch is already preserved by `finalize.test.ts:56–68` (direct `finalize()` `assert.throws` + hard_fail), so no seam coverage is lost by rewriting these.
   - **Resolution:** Add all of the above to the spec's ripple inventory. Rewrite `163–186`/`188–215` to assert `execute()` throws `CaptatumInputError(code:"extract_schema_unsupported_keyword")`; update `:264` to `reason:"transform_failed"`; fix `finalize.test.ts:10` (e.g. `reason:"unconfigured"`) and drop `reason` from the `mcp-shape.test.ts` success-transform fixture (reason is degrade-only). Re-grep for other `ModelRouterPort` fakes at implementation time. Stop claiming the narrowing is verified-clean.

### P2

3. **(P2 / message construction) The property-name segment of the path is not length-capped.** Spec lines 67–69 cap the unsupported **key** at ≤ 80 chars but leave the caller-controlled **property name** interpolated into the path uncapped, so an adversarial megabyte-long property name is echoed verbatim into the error/receipt — directly contradicting the spec's own stated principle ("an adversarial megabyte-long key must not bloat the error / receipt"). The path is built from caller property names (`json-schema.ts:167` `` `${path}.${key}` `` where `key` is the caller's property name); `bodyLimitBytes=5 MiB` (`config.ts:10`) bounds the worst case to a ~5 MiB error string in the JSON-RPC `error.message` / `structuredContent` (no explicit cap on error-message size exists).
   - **Resolution:** Also length-cap each property-name segment interpolated into the path (truncate ~80 chars + ellipsis), or cap the total composed path length. Apply the no-bloat principle to *both* caller-controlled strings echoed (key and property name), not just one.

4. **(P2 / contract precision) "The two never disagree" is imprecise — the walker is a deliberate SUPERSET of `validateAt`, and the two emit different PATHS for the same keyword.** `validateAt` (`json-schema.ts:28–51` + `json-schema-composites.ts:20–54`) recurses into `properties.*`, `items`, `additionalProperties`, `allOf`/`anyOf`/`oneOf`, `not` — but **never** into `$defs`/`definitions` (reference containers, never evaluated; captatum has no `$ref` support, so `$defs` content is currently dead/unreachable — making the new rejection *benign*, but still an undocumented behavior change for previously-accepted schemas). The walker explicitly recurses into `$defs.*/definitions.*` (spec lines 83–84), so a schema whose only unsupported keyword lives in `$defs` (e.g. `format`) is silently accepted by `finalize` today but newly **rejected** at input. Paths also differ: the walker reports schema-structural paths (`$.properties.email`) while `validateAt` reports data paths (`$.email`). All three lenses independently flagged this.
   - **Resolution:** State the divergence explicitly rather than claiming exact agreement: the per-node key-check is identical; the walker visits a superset of `validateAt`'s locations; the walker is intentionally stricter on `$defs`/`definitions` (fail-closed on unverifiable content at the trust boundary). Either document that `$defs`/`definitions` content is now validated (stricter) or drop them from the walker's recursion set to match `validateAt` exactly. Either is defensible; pick and say so.

5. **(P2 / scope / least-machinery) The closed 7-value `TransformReason` union exceeds issue #153's 3 values.** Spec lines 144–161 define 7 values; the issue lists 3 (`schema_validation_failed | transform_failed | unconfigured`). The 4 extras come from `noneReason` (`router-helpers.ts:27–37`) and are not a typed contract surface today. A minimal fix (add `schema_validation_failed`, rename `failed`→`transform_failed`, leave the rest as free strings) satisfies the issue with **no test-fake ripple**. The closed union is defensible under the allowlist principle but is deliberate scope expansion, not a forced consequence — and it is the root cause of the P1 test breakage.
   - **Resolution:** Either (a) keep the 7-value union but label it explicit scope expansion and pre-flag the test fixes (P1), or (b) narrow to the issue's 3 values + leave router sub-reasons as free strings to minimize ripple and respect least-machinery.

6. **(P2 / scope) The status-simplification is a real conformance fix, but scope beyond the literal issue — say so.** Verified: `noneReason` returns `no_model_fit`/`provider_unconfigured`/`model_unavailable`/`sensitive_content_no_local_provider`, which flow to `TransformInfo.reason` on a degrade via `rawFallback` (`router-helpers.ts:50`); the current allowlist `t.reason === "failed" || t.reason === "unconfigured"` at `shape.ts:76` and `bulk-seed.ts:35` misses them ⇒ `no_model_fit` degrade reports `status:"pass"`. Verified `provider:"none"` is set **only** on degrades (`captatum.ts:145,174,185,194`; explicit-raw returns at `:137` before transform is set; `rejectResult` at `:214` sets no transform), so `provider==="none"` ⇒ degrade is sound. Verified `contracts.md:484` already mandates `partial` when `transform.provider === "none"`, so this is a **fix**, not a contract change. The rename (`failed`→`transform_failed`) is what forces touching these lines; the simplification vs. just updating the strings is a judgment-call add.
   - **Resolution:** Accept as a conformance fix (contract already mandates `provider:none`). Note it as related scope, not literal issue scope.

7. **(P2 / walker under-specification) The walker's behavior on tuple-form `items` (JSON Schema array) is unspecified; C1–C6 do not cover it.** The spec says the walker "recurses into `items`" (line 83) without defining the array case. `validateAt` rejects tuple items entirely (`json-schema.ts:216` "schema items tuple arrays are not supported"); the `items` key itself is supported (`json-schema.ts:11`), so the node-level check passes and an implementer must decide whether to recurse into tuple elements (extra machinery) or skip.
   - **Resolution:** Specify: skip array-form `items` in the walker (tuple form is rejected at `finalize` anyway; recursing into elements is unrequested machinery).

8. **(P2 / sibling-sweep completeness) The sweep table and its "no other reader" claim miss the debug-text-channel reader of `TransformInfo.reason` in `format.ts:66`.** `src/interfaces/mcp/format.ts:66` — `` lines.push(`transform: …${t.reason ? ` (${t.reason})` : ""}…`) `` reads `result.transform.reason` for the debugTextBlock (MCP debug + CLI `--debug`). The spec's sweep table lists only `shape.ts` `classifyStatus` and `bulk-seed.ts` as reason readers; `format.ts:66` is a fourth reader. The rename silently changes the human/debug output from `(failed)` to `(transform_failed)` / `(schema_validation_failed)`. (Verified the audit log is NOT affected: the `ln.tool` event carries `transformProvider`/`Model`/`CostUsd`/`In`/`OutTokens`/`FallbackFrom` per `contracts.md:736`, no `reason`; and `bulk-result.ts:30` is a loose `reason?: string` envelope the spec correctly leaves as-is.)
   - **Resolution:** Add `format.ts:66` to the sweep table. No code change beyond the rename itself (display-only, renders the new value correctly), but the completeness claim should be accurate.

9. **(P2 / acceptance coverage) C2's "no fetcher invoked" assertion, as written, is trivially satisfied and proves nothing about the integration invariant.** C2 calls `normalizeCaptatumInput` directly — but `normalizeCaptatumInput` (`captatum-input.ts:100–120`) is a pure function with no fetcher parameter, so it *cannot* fetch by construction. This does not prove that `execute()` actually calls normalize *before* any fetch. The integration guarantee lives at `captatum.ts:65` (normalize) → `:68` (tier-2 short-circuit, which CAN fetch) → `:87` (Tier-1 fetch); a direct-normalizer test cannot witness that ordering.
   - **Resolution:** Add an `execute()`-level acceptance assertion with a fail-on-call fetcher mock (and a fail-on-call adapter) that confirms the `CaptatumInputError` reaches the caller with **zero** fetcher/adapter invocations. Unit-level (no browser); this is the honest proof of "before any fetch."

### Nits

10. **(nit / acceptance coverage) Two shipped behaviors are not exercised by C1–C6.** (a) The retained `finalize` defense-in-depth branch producing `reason:"schema_validation_failed"` (spec lines 115–120) is never asserted at acceptance; (b) `outputRequested` on a **non-degraded** (successful) result is never asserted, even though the spec stamps it on "every agent-facing Result." `computeProvenanceHash` (`result.ts:135–148`) excludes `outputRequested`, so no hash ripple — these are pure coverage gaps.
    - **Resolution:** Add (a) a unit test calling `finalize` directly with an unsupported-keyword schema asserting `reason:"schema_validation_failed"`; (b) a test that a successful summary carries `outputRequested:"summary"`. Both cheap; lock in stated behavior.

11. **(nit / spec accuracy) The retained `finalize` unsupported-keyword branch is dead in the production call graph; "defense-in-depth" rests on a speculative future caller.** The only production `TransformPort` caller is `captatum.ts:159` (downstream of normalize; `src/dev/smoke-test.ts:37` is dev-only). After design (2), `request.schema` is guaranteed clean at every `validateAt`-visited location, so `finalize`'s `validateSupported` (`finalize.ts:49,52–58`) is unreachable in production. Keeping it is defensible (existing fail-closed seam; least-machinery favors not deleting), but `contracts.md:578` "on the rare reach" overstates it.
    - **Resolution:** Acceptable to keep; state plainly it is dead in the current call graph, retained only for hypothetical direct-`TransformPort` callers.

12. **(nit / spec accuracy) `outputRequested` "stamped on every agent-facing Result" is overstated.** Bulk synthetic error results (`bulk-seed.ts` `abortedSeedResult`/`wallAbandonedResult`/`syntheticFail`) set `output:"raw"` with no `outputRequested`; they are `tier:error`/`fail` so `outputRequested` adds nothing, but spec lines 134–136 overstate coverage — it is stamped only on the `applyOutputMode` + `rejectResult` paths.
    - **Resolution:** Clarify `outputRequested` is stamped on `applyOutputMode` + `rejectResult` only (not synthetic bulk error results).

13. **(nit / spec accuracy) "the four `noneReason` values" should be "five."** Spec line 158 says "the four `noneReason` values (`router-helpers.ts:27–37`)" but `noneReason` returns **five** (`unconfigured`, `sensitive_content_no_local_provider`, `provider_unconfigured`, `model_unavailable`, `no_model_fit`). The spec's own union at lines 145–153 lists all five correctly, and `contracts.md:584` enumerates all seven — so this is cosmetic; the enum and contract are correct.
    - **Resolution:** Change "four" to "five" in the spec prose.

## Notes

- **The core security posture is sound.** The walker is a verified sound superset of `validateAt`'s traversal; the allowlist-at-boundary + fail-closed + no-value-echoed design is correct; both `normalize*` throw before egress and are wrapped to `InvalidParams`. The status-classification latent bug is real and the `provider:"none"`⇒degrade invariant is provably sound (4 writers verified, explicit-raw returns before transform is set). `contracts.md` is already consistent.
- **The block is narrow and cheap.** A depth cap is ~3 lines and a wording correction; it does not touch the design's shape. The P1 is pure documentation (enumerating tests the coder must already fix to get green). Neither requires reopening a design decision — only correcting two false claims (depth bound; verified-clean narrowing) and stating one coverage delta the spec currently paper- over.
- **Severity split on the depth cap is itself signal.** All three lenses flagged the same underlying defect; they differ only on whether JSON.parse's implicit recursion limit is an adequate control today (correctness-security: yes, RangeError is caught → `InternalError`, no crash vector) or an unnamed residual on a tier-S trust boundary (contract-scope: blocker; siblings-tests: most-likely-to-draw-review-fire). The aggregate adopts the blocker reading consistent with the repo's "fail-closed on untrusted input at trust boundaries" rule.
