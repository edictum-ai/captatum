# Critique — spec for #157 (bulk wall env knob)

Independent synthesis of three lens-critics (security / completeness / process), with every
claim re-verified against the actual source in this worktree. One lens-critic finding was
materially **wrong** about the current source (see the note under the verify-bar row) and is
recorded as wrong-with-reason so the implementer does not act on a false premise.

---

## 1. Verdict

The spec is **sound to implement** after addressing the findings below. There are **no blockers**:
the design (one env knob → `config.ts` accessor with strict regex + bound check → hosted wiring,
with an optional local honor-the-env completion) is correct, the contract/threat-model docs are
already updated in lockstep, and the fail-closed semantics are right. The open items are **spec-text
precision fixes plus one genuine scope decision**, not re-architecture.

The decisions that MUST be closed before coding (all resolvable in the spec, none needing new
investigation):

1. **Local honor-the-env vs hosted-only.** §2 proposes the local-binary flavor also read
   `CAPTATUM_BULK_MAX_GLOBAL_WALL_MS` (a "consistency completion" beyond the issue comment's literal
   "hosted only"). That is a real behavior change (a hosted runbook value copied to local silently
   *narrows* the local wall 180 s→55 s), and §2's "byte-identical / no-regression" framing is only
   true when the env is unset. Decide: (a) drop §4.3 (hosted-only, least machinery, literal to the
   issue), or (b) keep it and reframe + add a warning.
2. **Capture placement.** §4.2 elides where `const bulkMaxGlobalWallMs = config.bulk.maxGlobalWallMs()`
   sits relative to the `config.bulk.enabled()` ternary. Pin it *inside* the truthy branch so a
   malformed value cannot boot-reject a bulk-**disabled** hosted server — or explicitly decide
   validate-always and reconcile that with §6's "avoid boot-breakage of deployments with stray env
   vars" rationale for the sibling knobs.
3. **Boot-reject vs clamp+warn.** The spec's choice (boot-reject) is defensible under the literal
   fail-closed rule, but it takes the whole process down on an operator typo. Acknowledge the
   availability cost and the rejected clamp+warn alternative so the choice is visible, not implicit.

One process-lens finding ("verify bar is impossible: no wall value in any bulk receipt") is
**wrong** and is flagged as such: `BulkResult.guard.maxGlobalWallMs` IS in every bulk receipt
(verified: `src/domain/bulk-result.ts:76`, `src/domain/bulk-policy.ts:17`,
`src/application/use-cases/bulk-assemble.ts:62`, `captatum-bulk.ts:91`). The real, smaller point
inside that finding — that `pnpm run smoke` never calls `captatum_bulk` — stands and is kept.

## 3. Open decisions for the implementer (resolve before coding)

1. **Local flavor: honor the env, or hosted-only?** (rows 3) — The issue comment says hosted-only;
   §2/§4.3 propose local honor-the-env as a "consistency completion." Decide (a) drop §4.3 (least
   machinery, literal to the issue, local keeps its 180 s via the hardcoded ceiling) or (b) keep it
   and reframe §2 as a behavior change + add the runbook-narrows-local warning to contracts.md.
   Recommendation: (a) — the domain already gives local its 180 s wall; adding an env the local
   operator has no reason to set introduces a narrowing footgun for no real local benefit.
2. **Capture placement: validate only when bulk is enabled, or always?** (row 1) — Pin inside the
   `config.bulk.enabled()` true branch, consistent with §6's avoid-boot-breakage stance. Must be
   stated in §4.2, not left to the implementer's reading of "capture once."
3. **Boot-reject vs clamp-to-default + loud-warn: confirm the tradeoff.** (row 12) — The spec chose
   boot-reject; that is the strict fail-closed reading. Confirm that is intended given the
   whole-process availability cost on an operator typo, and that the asymmetry vs the silent-clamp
   sibling knobs is deliberate (this is the DoS wall; they are not). No code change expected; one
   sentence in §5.

All three are resolvable by editing the spec text. None requires re-running the lens critics or
re-opening the contract.
